# multi_game_portal

## 見てほしいポイント

* モンスター詳細ページ（UI構造）
* ログイン機能（UX）

---

## 開発の背景・目的

攻略サイトは1ページあたりの情報量が多く、目的の情報にたどり着くまでにスクロールやページ遷移が多くなるという課題があると考えました。
そのため、ユーザーが必要な情報へ最短でアクセスできるUIと情報構造を実現することを目的として開発しました。

---

## 概要

Djangoを用いて開発したゲーム攻略サイトです。
モンスター、アイテム、マップ、ボス攻略など複数カテゴリの情報を扱い、ユーザーが必要な情報へ素早くアクセスできる構成を意識して設計しました。

---

## 技術選定理由

バックエンドにはDjangoを採用しました。認証機能や管理画面が標準で提供されており、ユーザー管理やコンテンツ管理を効率的に実装できると考えたためです。
また、攻略サイトのようにコンテンツ数が増えるサービスでは、管理画面からデータを更新できる点が適していると判断しました。

## 開発体制

本プロジェクトは3名のチームで開発を行い、私は主にフロントエンドを担当しました。

UI設計、テンプレート作成、CSS設計、JavaScriptによるインタラクション実装を担当し、情報量の多い攻略サイトにおいてユーザーが迷わず操作できる画面設計を意識しました。

特に、サイドバー目次やアンカーリンク、モーダル表示、ホバーによる情報提示などを実装し、ユーザーが必要な情報へスムーズにアクセスできるUIを意識しました。

また、バックエンド側のデータ構造を踏まえながらテンプレート設計を行い、表示ロジックとUIの分離を意識した実装を行いました。

---

## 主な機能

* Django認証機能を利用したログイン / 新規登録 / パスワードリセット
* コメント機能によるユーザー参加型の情報共有
* モンスター、アイテム、マップ情報の一覧・詳細表示
* サイドバー目次とアンカーリンクによるページ内ナビゲーション
* 画像クリックによるモーダル表示
* ホバーによる補足情報の段階的表示

---

## 工夫した点

* 攻略サイトは情報量が多く、スクロールのみでは目的の情報に到達しづらいという課題があるため、サイドバー目次とアンカーリンクを実装し、弱点・ドロップ・出現場所などの各セクションへ直接移動できる構成にしました。

* すべての情報を一度に表示すると視認性が低下するため、画像はモーダル表示、補足情報はホバー表示に分け、必要なタイミングで情報を確認できるUIにしました。

* モンスター・アイテム・マップなど複数ページで見出し構造や情報配置を統一し、ページごとに操作方法を覚え直す必要がないように設計しました。

* Djangoの認証機能を利用し、ログイン・新規登録に加えてパスワードリセット導線まで実装することで、閲覧だけでなく継続利用を想定した設計にしました。

---

## 課題

* ページ単位で実装を進めたため、views.py と urls.py に個別ページごとの処理やルーティングが集中し、保守しにくい構成になりました。特に、URL設計を slug ベースで統一せず個別定義を増やしたことで、機能追加時の修正コストが高くなった点が課題です。

* 一部のデータについてはDjangoのモデルを用いて管理していますが、全体としてはページごとの実装が中心となっており、データ駆動の設計には十分にできていない点も課題です。

* HTML / CSS の重複やテンプレートの個別作成が増えたことで、共通部分の修正範囲が広がり、保守性が低下しました。

---

## 学んだこと

* 情報量の多いサービスでは、UIの見た目だけでなく「目的の情報へどれだけ早く到達できるか」という導線設計が重要であること

* 機能追加を優先すると、views や urls に責務が集中しやすく、設計を後から修正するコストが大きくなること

* slug を用いたURL設計やモデル設計を先に整理し、データを中心に設計することで拡張性と保守性を向上できること

* ページ数が増えるほど、テンプレートの共通化や再利用性を意識した設計が不可欠であること
